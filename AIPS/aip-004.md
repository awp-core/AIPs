## Abstract

This AIP specifies KYA WorkNet, the agent identity infrastructure for the AWP ecosystem and beyond. KYA aggregates identity signals — social claims, on-chain economic commitments, and biometric verification — into a standardized attestation layer queryable by any agent network. KYA does not make access decisions; it provides the data, and consumers define their own trust policies. The proposal defines attestation types, verification workflows, token economics ($aKYA with linear 2-year emission), a seven-category distribution model, burn-on-query and stake-to-serve utility mechanics, and an extensibility framework for third-party attestation providers. KYA deploys on Base with the WorkNet token `$aKYA` tradable against `$AWP` via Uniswap V4.

## Motivation

### 51% of the Internet Is No Longer Human

In 2025, bots and AI agents surpassed human beings as the majority of internet traffic for the first time — 51% and accelerating, with autonomous agent traffic growing 7,851% year-over-year. BNB Chain alone has 122,000+ on-chain AI agents. Gartner projects 40% of enterprise applications will feature AI agents by end of 2026, up from under 5% a year prior. The agent economy is not a forecast — it is measured, reported, and growing exponentially.

This economy is operating without an identity system.

### The Trust Vacuum

Every economy that scales needs identity infrastructure. The human internet built credit bureaus, domain registrars, SSL certificate authorities, and OAuth providers over two decades. These layers are invisible and unglamorous. Without them, e-commerce, online banking, and the $3 trillion digital economy would not exist.

The agent economy needs its own versions of these layers. And it needs them now — because agents are arriving a thousand times faster than human users ever did.

An AI agent today is an Ethereum address. No name, no face, no history, no accountability. You cannot distinguish a well-funded agent with 10,000 AWP staked and a year of verified work history from a sybil clone spawned five minutes ago to exploit a reward system. The signals exist — staking records, social claims, biometric verification, work histories — but they are scattered across chains, platforms, and protocols with no unified query layer.

The result: AI-enabled crypto fraud hit $17 billion in 2025. Sybil attacks routinely compromise agent networks where 70% of "active wallets" turn out to be controlled by a handful of operators. Enterprise adoption stalls because teams share human credentials with agents — no alternative identity model exists. NIST launched a formal initiative in February 2026 to address agent identity governance. When the U.S. federal government scrambles to define the problem, the gap is real.

### Every Network Reinvents the Wheel

Every agent network today builds its own ad hoc identity system. Mine WorkNet (AIP-1) maintains a credit score system, AI Proof-of-Work challenges, and IP decay enforcement. Predict WorkNet (AIP-2) has rate limiting and per-timeslot caps. Every new WorkNet that launches will face the same sybil question and build yet another bespoke solution.

This is the equivalent of every website in 1998 inventing its own encryption protocol — wasteful, fragile, and guaranteed to fail at scale.

### Why KYA

KYA replaces per-project identity overhead with shared infrastructure. One API call — `GET /v1/agents/{address}/attestations` — returns everything known about an agent: social identity, economic commitment, human verification, work history. Each consumer defines its own trust policy on top of the same data layer.

The design philosophy is deliberate: **KYA does not make access decisions. It provides attestation data. Consumers decide.** This is how the most powerful identity infrastructure in the traditional world operates. Equifax does not decide whether you get a mortgage — it provides the data, and the bank decides. DNS does not determine whether you trust a website — it resolves the address, and your browser checks the certificate. KYA is the neutral ground for agent identity.

### Why AWP, and Why a WorkNet

AWP gives KYA what no greenfield startup has: **92,526 agents already registered and working.** That is not a waitlist. It is 92,526 agent addresses submitting work, getting scored, earning token rewards, and interacting with WorkNet infrastructure every day. Each one needs identity verification. Each one is a day-one user.

Cold-start kills identity systems. Every identity network faces the same chicken-and-egg problem: users will not register until the network is useful, and the network is not useful until users register. KYA solves this by launching inside a live ecosystem where demand is already saturated — 92,526 agents across multiple WorkNets, every one needing trust signals, every WorkNet building its own ad hoc solution.

As a WorkNet, KYA is woven into AWP's economic engine. It receives AWP emission to fund the bootstrap phase — incentivizing attestation registration, rewarding verification work, subsidizing consumer integration. And the flywheel runs both ways: KYA makes every other WorkNet stronger by replacing their identity overhead with one API call. Other WorkNets run more efficiently, the ecosystem gets healthier, veAWP holders see more value, and they vote for more emission to KYA.

From AWP, KYA expands outward. Nothing about the attestation schema is AWP-specific. Once KYA has the densest agent identity graph on-chain, external platforms integrate it because the data is there.

| Liquidity  | $aKYA ↔ $AWP on Uniswap V4 (Base)                            |

At activation, the AWP protocol mints 1,000,000,000 $aKYA (10% of max supply) and pairs it with 1,000,000 $AWP (escrowed at registration) to bootstrap the Uniswap V4 liquidity pool. This is a one-time protocol-level operation, separate from the emission schedule.

The remaining 9,000,000,000 $aKYA is minted by the WorkNet Manager according to the linear schedule in §1.1. No pre-mine, no team allocation.

#### 1.1 Emission Schedule

KYA uses a linear emission model: equal daily emission over 730 days (2 years), after which emission ceases.

```
daily_emission = 9,000,000,000 / 730 = 12,328,767 $aKYA
```

| Parameter                  | Value                            |
| -------------------------- | -------------------------------- |
| Max supply                 | 10,000,000,000 $aKYA             |
| Initial LP (at activation) | 1,000,000,000 $aKYA (10%)        |
| Emittable supply           | 9,000,000,000 $aKYA (90%)        |
| Emission duration          | 730 days (2 years)               |
| Daily emission             | 12,328,767 $aKYA                 |
| Monthly emission           | ~375,000,000 $aKYA               |
| Year 1 total               | 4,500,000,000 $aKYA (45% of max) |
| Year 2 total               | 4,500,000,000 $aKYA (45% of max) |
| Post-730 days              | Zero emission                    |

**Rationale for linear emission.** Identity infrastructure is a winner-take-most market where speed of network effect accumulation determines the outcome. Exponential decay or halving models front-load emission but taper off, reducing bootstrap incentives over time. Linear emission maintains consistent incentive pressure for the full 2-year bootstrap window, then stops cleanly. KYA's value after year 2 is sustained by utility demand (query burns, staking requirements) rather than emission — the identity graph either has critical mass by then, or it does not.

### 2. Token Utility

$aKYA has four utility mechanics that create sustained demand beyond emission:

#### 2.1 Query Burn

Consumers querying the KYA API pay in $aKYA. Payment is **burned** — permanently removed from circulating supply.

| Tier     | Query Type                                             | Cost          | Burn        |
| -------- | ------------------------------------------------------ | ------------- | ----------- |
| Free     | Basic attestation lookup (≤10 queries/day per API key) | 0             | —           |
| Standard | Unlimited queries, real-time data                      | Per-query fee | 100% burned |
| Bulk     | Historical data, batch queries, analytics              | Volume-priced | 100% burned |

Query pricing is set by the Owner and MAY be adjusted with 7 days notice. Free tier exists to lower integration friction during bootstrap; the Owner MAY reduce or eliminate the free tier as adoption grows.

**Burn accounting**: all burns are executed on-chain via a `burn(amount)` call on the $aKYA token contract, producing auditable `Transfer(from, address(0), amount)` events.

#### 2.2 Service Provider Staking

Third parties building services on KYA data (lending protocols, credit scoring services, compliance platforms, agent hiring services) MUST stake $aKYA to access the Service Provider API tier.

| Parameter        | Value                                              |
| ---------------- | -------------------------------------------------- |
| Minimum stake    | Set by Owner (initial: 100,000 $aKYA)              |
| Lock duration    | Minimum 90 days                                    |
| Slash conditions | Misuse of KYA data, violation of data access terms |
| Unstake notice   | 30 days                                            |

Staking signals economic commitment and aligns service providers with the KYA ecosystem. Staked $aKYA is held in the WorkNet Manager contract.

#### 2.3 Attestation Provider Staking

Third-party verification providers who submit attestations to KYA (security auditors, domain expertise certifiers, specialized compliance checkers) MUST stake $aKYA.

| Parameter        | Value                                         |
| ---------------- | --------------------------------------------- |
| Minimum stake    | Set by Owner (initial: 50,000 $aKYA)          |
| Lock duration    | Minimum 180 days                              |
| Slash conditions | Submitting false or unverifiable attestations |
| Unstake notice   | 30 days                                       |

This creates a quality gate for the attestation marketplace: providers put capital at risk as a guarantee of attestation integrity.

#### 2.4 Dispute Resolution

When an attestation is challenged:

1. Challenger stakes $aKYA (minimum: 1,000 $aKYA)
2. Attestation enters `disputed` status
3. Verification agents investigate and submit evidence
4. Owner adjudicates (Phase 1); community governance adjudicates (future)
5. Outcome: challenger wins → attestation revoked, attestation provider's stake slashed, challenger refunded + reward. Challenger loses → challenger's stake forfeited to verification agent pool.

This ensures challenges have economic cost (preventing spam) while providing recourse against fraudulent attestations.

### 4. Attestation System

KYA's core product is the attestation layer. Each attestation is a verifiable claim about an agent, backed by cryptographic proof and queryable through a standardized API.

#### 4.1 Attestation Schema

All attestations share a common structure:

```json
{
  "id": "att_<unique_id>",
  "agent_address": "0x...",
  "type": "<attestation_type>",
  "status": "active | expired | revoked | pending | disputed",
  "created_at": "<ISO 8601>",
  "updated_at": "<ISO 8601>",
  "expires_at": "<ISO 8601 | null>",
  "proof": {
    "type": "<proof_type>",
    "value": "<verifiable_reference>"
  },
  "metadata": {}
}
```

The schema is designed for extensibility: new attestation types add new `type` values and `metadata` fields without changing the core structure. W3C DID format compatibility is maintained for interoperability with the broader decentralized identity stack.

#### 4.2 MVP Attestation Types

Three attestation types at launch, each attacking a different dimension of trust:

**Type 1: Social Identity (twitter_claim)**

An agent's owner posts a cryptographically signed claim on Twitter linking their social identity to the agent's address.

```json
{
  "type": "twitter_claim",
  "proof": { "type": "tweet_url", "value": "https://twitter.com/alice_web3/status/..." },
  "metadata": {
    "twitter_handle": "alice_web3",
    "twitter_uid": "98765432",
    "tweet_id": "1234567890",
    "claim_text": "I am the owner of agent 0xAbC...123 on AWP #KYA"
  }
}
```

Verification workflow:

1. Agent requests a claim challenge from KYA (signed with agent wallet via EIP-712)
2. KYA generates a unique claim text containing the agent address and a nonce
3. Agent's owner posts the claim text as a tweet
4. Agent submits the tweet URL to KYA
5. KYA verifies via Twitter API v2: tweet exists, content matches, account is active
6. Verification passed → attestation status: `active`

Social attestations do not expire by default. Future re-verification (checking tweet still exists) MAY be added.

**Type 2: Economic Commitment (staking)**

KYA indexes on-chain staking data in real time — balances, lock durations, veAWP positions. No user action required; KYA's staking sync service monitors the chain automatically.

```json
{
  "type": "staking",
  "proof": { "type": "tx_hash", "value": "0x..." },
  "metadata": {
    "chain_id": 8453,
    "contract_address": "0x0000b534C63D78212f1BDCc315165852793A00A8",
    "staked_amount": "10000000000000000000000",
    "staked_amount_display": "10,000 AWP",
    "lock_end_time": "2026-10-21T00:00:00Z",
    "last_synced_at": "2026-04-21T10:30:00Z"
  }
}
```

Sync mechanism:

- Event-driven: listen to veAWP contract `Deposit`, `Withdraw`, `PartialWithdraw` events for real-time updates
- Polling fallback: every 10 minutes, full-scan known agent addresses
- Status follows chain state: active stake → `active`; expired/withdrawn → `expired`
- `last_synced_at` provided so consumers can assess data freshness

Economic signals are the hardest to fake. A million Twitter accounts cost nothing; 50,000 AWP locked on-chain is real capital at risk.

**Type 3: Human Verification (kyc)**

Through Didit integration, an agent's owner completes face + document verification. KYA stores only the verification result — never biometric data or PII. The KYC provider is abstracted behind a `KycProvider` interface, allowing future migration to alternative providers (Sumsub, Persona, etc.) without architectural changes.

```json
{
  "type": "kyc",
  "proof": { "type": "provider_ref", "value": "didit_session_abc123" },
  "metadata": {
    "provider": "didit",
    "owner_address": "0xOwner...456",
    "verification_level": "document_and_selfie",
    "country": "US",
    "verified_at": "2026-04-21T09:30:00Z"
  }
}
```

Verification workflow:

1. Agent initiates KYC request (signed with agent wallet)
2. Owner approves the request (signed with owner wallet)
3. Dual-signature verification: both agent and owner cryptographically prove their respective addresses
4. Owner redirected to Didit for face + document verification
5. Didit webhook returns result to KYA
6. Verification passed → attestation status: `active`; owner-agent binding recorded

KYA stores: verification status (pass/fail), provider reference ID, country. KYA does NOT store: photos, document images, biometric data, names, dates of birth, or any other PII. Didit handles all PII under its own GDPR-compliant data policy.

KYC attestations expire after 1 year and require re-verification.

### 5. Verification Protocol

Verification agents perform ongoing attestation quality assurance. This is the "work" that earns the 10% Verification Agent allocation.

#### 5.1 Verification Tasks

| Task Type                | Description                                                  | Frequency                                                |
| ------------------------ | ------------------------------------------------------------ | -------------------------------------------------------- |
| Twitter re-check         | Verify claimed tweets still exist and content unchanged      | Weekly sample (10% of active twitter_claim attestations) |
| Staking cross-validation | Independent on-chain read to confirm staking sync accuracy   | Daily sample (5% of active staking attestations)         |
| Anomaly detection        | Flag suspicious patterns: same owner claiming 100+ agents, attestation metadata inconsistencies | Continuous                                               |
| Dispute investigation    | When a challenge is filed (§2.4), gather evidence and submit findings | On-demand                                                |

#### 5.2 Verification Agent Requirements

- Registered with KYA as a verification agent
- MUST have an active staking attestation (minimum 1,000 AWP)
- MUST maintain ≥ 90% accuracy on verification tasks (measured against ground truth re-checks by KYA Coordinator)

#### 5.3 Verification Reward Calculation

```
v_reward(agent) = (tasks_completed × accuracy_score) / Σ(tasks_completed × accuracy_score)
                  × verification_pool
```

Rewards are proportional to work volume weighted by accuracy. Low-accuracy verification agents earn disproportionately less per task.

### 6. Consumer Integration

#### 6.1 Query API (Public)

```
GET /v1/agents/{agent_address}/attestations
    ?type=twitter_claim,staking,kyc
    &status=active
```

Returns all matching attestations for the given agent address. Consumers apply their own filtering and trust policies.

```
GET /v1/agents/{agent_address}/trust-score
```

Returns a composite trust score (see §7) — a single number summarizing the agent's attestation portfolio.

```
GET /v1/owners/{owner_address}/agents
```

Returns all agents bound to an owner via KYC attestation.

#### 6.2 Submission API (Authenticated, EIP-712)

```
POST /v1/attestations/twitter/claim    — Initiate Twitter claim
POST /v1/attestations/kyc/init         — Initiate KYC verification
POST /v1/attestations/custom/submit    — Third-party attestation submission (requires Attestation Provider staking)
```

All submission endpoints require EIP-712 signature proving control of the agent address.

#### 6.3 Service Provider API (Staked)

Requires Service Provider staking (§2.2). Provides:

- Bulk query endpoints (batch lookup by address list)
- Real-time webhook feeds (new attestation, status change, dispute events)
- Historical attestation data and analytics
- Trust score computation API with custom weighting

### 7. Trust Score

With sufficient attestation data, KYA computes a composite trust score per agent — the agent equivalent of a credit rating.

#### 7.1 Score Composition

```
trust_score = Σ (attestation_type_weight × attestation_value)
```

| Attestation Type      | Weight | Value Calculation                               |
| --------------------- | ------ | ----------------------------------------------- |
| twitter_claim         | 10     | 10 if active, 0 otherwise                       |
| staking               | 0–40   | Proportional to staked amount and lock duration |
| kyc                   | 30     | 30 if active, 0 otherwise                       |
| work_history (future) | 0–20   | Based on verified work across WorkNets          |

Maximum score: 100. Published methodology, deterministic computation from public inputs.

#### 7.2 Score Properties

- **Deterministic**: same attestation portfolio → same score, always
- **Public formula**: published in skillsURI, version-controlled
- **Non-gatekeeping**: the score is informational. KYA does not use it to restrict access. Consumers decide their own thresholds.
- **Extensible**: new attestation types add new score components without changing the formula structure

### 8. Transparency and Auditability

KYA commits to the same transparency standard required of all AWP WorkNets (Builder Guide §3.3):

#### 8.1 On-Chain Checkpoints

```solidity
// Each epoch, commit attestation activity and distribution data
manager.submitCheckpoint(epoch, merkleRoot, uri);
```

The Merkle root covers: epoch attestation counts by type, verification task results, reward distribution CSV (address, amount, category), and aggregate query volume.

#### 8.2 Published Data

Each epoch, KYA publishes:

- Distribution CSV: all reward recipients with amounts and category labels
- Attestation activity summary: new registrations, verifications, disputes, expirations
- Query volume metrics (aggregate, not per-consumer)
- Burn amount (total $aKYA burned via queries in the epoch)

Published at the KYA skillsURI repository and linked from the on-chain checkpoint URI.

### 9. Data Privacy

#### 9.1 Core Principle

KYA is an attestation aggregator, not a data custodian. KYA stores **verification results**, not underlying identity data.

| Data                             | Stored by KYA                        | Stored by Third Party |
| -------------------------------- | ------------------------------------ | --------------------- |
| Twitter handle + tweet URL       | Yes                                  | Twitter               |
| Staking balances + lock times    | Yes (indexed from public chain data) | Public blockchain     |
| KYC result (pass/fail) + country | Yes                                  | Didit (PII custodian) |
| Photos, documents, biometrics    | **No**                               | Didit                 |
| Names, dates of birth, addresses | **No**                               | Didit                 |

#### 9.2 Compliance

- W3C DID format compatibility for interoperability with decentralized identity standards
- GDPR: KYA stores no PII directly. Didit is the data processor for KYC biometrics under its own GDPR compliance.
- Right to erasure: agents can request attestation deletion. KYA removes the attestation record; the on-chain checkpoint Merkle root remains (it commits to aggregate data, not individual records).

### 12. Security Considerations

#### Sybil Attacks on Attestation Registration

**Attack**: create thousands of agent addresses, register minimal attestations on each, collect 10% emission.

**Mitigations**:

- Twitter claims require unique Twitter accounts (one-to-one binding)
- KYC requires human verification (biometric uniqueness)
- Staking attestations require real capital commitment
- Registration rewards are weighted by attestation quality (KYC > staking > Twitter), not just count
- Verification agents continuously monitor for anomalous patterns (same owner claiming excessive agents)

#### False Attestation Injection

**Attack**: attestation provider submits fabricated attestations to inflate agents' trust scores.

**Mitigations**:

- Attestation providers must stake $aKYA (minimum 50,000); false attestations → slash
- Verification agents independently re-check attestation claims
- Dispute mechanism allows any party to challenge an attestation with economic stake
- All attestations carry verifiable proofs (tweet URLs, tx hashes, provider references) that can be independently checked

#### Coordinator Trust

KYA's Coordinator is centralized at launch. Mitigations mirror the AWP protocol standard:

- **No fund custody**: all token operations are on-chain via WorkNet Manager
- **Verifiable proofs**: every attestation includes a proof field that can be independently verified without trusting the Coordinator
- **On-chain checkpoints**: epoch data committed on-chain via Merkle roots
- **Published distributions**: all reward allocations published as CSVs, verifiable against checkpoints
- **Progressive decentralization**: §11 roadmap moves verification and governance off the centralized Coordinator

#### Economic Attack on Query Burns

**Attack**: front-run or manipulate query pricing to extract value.

**Mitigation**: query prices are fixed per epoch and published in advance. Burns are on-chain and auditable. Price changes require 7 days notice.

### 13. Future Extensions

The following are not part of this specification but are planned directions:

**Work history attestations.** WorkNets submit verified work records to KYA: tasks completed, quality scores, epochs participated. Every agent gets a cryptographically verified résumé that travels across WorkNets.

**Cross-chain identity.** Index staking and activity across all AWP-supported chains (Base, Ethereum, Arbitrum, BSC) and external protocols into a unified agent profile.

**Trust score boosting.** Agents stake $aKYA directly on their KYA profile as an additional trust signal — not buying a score, but adding economic commitment specific to their KYA identity.

**Agent-to-agent trust handshake.** A standardized protocol where agents query each other's KYA profiles before transacting — machine-readable trust at internet scale.

## Backwards Compatibility

This is a new WorkNet with no existing state. No backwards compatibility concerns. KYA introduces no changes to the AWP protocol, emission mechanism, or governance system — it operates entirely within the existing WorkNet framework.
